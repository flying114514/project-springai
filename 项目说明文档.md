# 项目说明文档

## 1. 项目一句话介绍
这是一个面向求职场景的 AI 面试辅助平台，围绕 **简历分析、模拟面试、知识库问答、RAG 对话** 四条核心链路，帮助用户完成“上传资料 → AI 分析 → 训练表达 → 输出报告”的闭环。

它不是一个单点的聊天机器人项目，而是一个把 **LLM 调用、向量检索、文件处理、对象存储、缓存恢复、异步任务、结果导出** 串起来的完整后端工程。

---

## 2. 项目定位与要解决的问题

### 2.1 项目定位
从业务视角看，这个项目更像一个“AI 求职助手后端”：
- 对简历做结构化分析与评分
- 根据简历生成模拟面试题
- 对用户答案进行评估并产出面试报告
- 支持上传知识库文件做检索增强问答
- 支持多知识库 RAG Chat 会话，形成持续问答场景

### 2.2 解决的核心问题
传统求职准备往往有几个痛点：
1. **简历写得出来，但不知道哪里强、哪里弱**
2. **会做项目，但面试表达不够结构化**
3. **资料很多，但临时问答很难快速定位信息**
4. **AI 返回不稳定，难直接接入工程系统**

这个项目的目标，就是把这些问题拆成几个工程能力：
- 用 **结构化 AI 输出** 解决“结果不稳定”
- 用 **Redis Stream 异步化** 解决“AI 耗时长”
- 用 **PostgreSQL + pgvector** 解决“知识库检索”
- 用 **对象存储 + 文档解析** 解决“文件上传与长期管理”
- 用 **PDF 导出** 解决“结果交付”

### 2.3 适合面试时怎么定义这个项目
可以这样讲：

> 这是一个基于 Spring Boot 和 Spring AI 的 AI 求职辅助平台。它不是只做一次 LLM 调用，而是把简历上传、文档解析、知识库向量化、模拟面试、异步评估、RAG 检索和 PDF 报告导出整合成了一套完整后端系统。重点不只是“用了 AI”，而是把 AI 能力工程化、产品化。

## 4. 整体架构设计

### 4.1 分层思路
整个项目基本是典型的分层后端架构，但加入了 AI 工程化能力：

1. **Controller 层**
   - 接收请求、校验参数、组织返回值
   - 对外暴露简历、面试、知识库、RAG Chat 等接口

2. **Service 层**
   - 承担业务编排
   - 串联文件处理、AI 调用、缓存、数据库、异步任务
   - 是项目的核心复杂度所在

3. **Infrastructure 层**
   - 封装底层通用能力
   - 包括文件存储、文档解析、Redis、PDF 导出、对象映射等
   - 好处是业务层不直接依赖底层细节，复用性更强

4. **Repository / Entity 层**
   - 使用 JPA 访问 PostgreSQL
   - 存储简历、面试会话、面试答案、知识库、RAG 消息等数据

5. **外部依赖层**
   - DashScope / OpenAI 兼容接口：大模型能力
   - PostgreSQL + pgvector：业务数据和向量数据
   - Redis：缓存、限流、异步任务
   - S3 兼容对象存储：原始文件持久化

### 4.2 架构图（面试时可口述）
```text
前端 / 调用方
   ↓
Controller 层
   ↓
Service 层（业务编排）
   ├─ AI 调用（Spring AI + DashScope）
   ├─ Redis 缓存 / Redis Stream 异步任务
   ├─ 文档解析 / 对象存储 / PDF 导出
   └─ Repository 持久化
        ↓
PostgreSQL / pgvector / Redis / S3 兼容对象存储
```

### 4.3 这个架构的价值
这个分层的价值不是“看起来规范”，而是解决真实问题：
- Controller 层保持薄，避免请求逻辑和业务逻辑混杂
- 业务复杂度集中在 Service 层，便于模块化扩展
- 底层能力沉到 Infrastructure，便于跨模块复用
- AI 调用、向量检索、文件处理这些易变部分被封装，后续替换成本更低

## 5. 核心模块说明

## 5.1 简历模块

### 模块职责
简历模块负责：
- 上传简历
- 校验文件类型与大小
- 提取正文文本
- 调用 AI 做简历评分与建议输出
- 保存分析历史
- 导出简历分析报告 PDF
- 支持重复简历复用历史结果

### 关键类
- `ResumeController`
- `ResumeUploadService`
- `ResumeGradingService`
- `DocumentParseService`
- `FileStorageService`
- `PdfExportService`

### 实现思路
1. 前端上传简历后，先做文件大小和类型校验
2. 尝试识别是否为重复简历，避免重复分析
3. 用 Tika 提取文本内容
4. 原文件上传到对象存储，文本与元数据保存到数据库
5. 分析任务进入 Redis Stream，异步执行 AI 评分
6. AI 评分结果以结构化 JSON 方式解析为 Java 对象
7. 最终支持历史查看、详情查看和 PDF 导出

### 这个模块的面试亮点
- 不是简单地“上传文件然后调一下 AI”，而是做了 **去重、异步化、结构化输出、结果持久化、报告导出** 的完整闭环
- AI 分析失败时也不是直接崩，而是有统一错误处理与重试策略

---

## 5.2 模拟面试模块

### 模块职责
模拟面试模块负责：
- 创建面试会话
- 基于简历生成问题
- 管理当前进行到第几题
- 支持暂存答案与提交答案
- 会话中断后从数据库恢复
- 完成后异步评估答案并生成报告
- 导出模拟面试 PDF 报告

### 关键类
- `InterviewController`
- `InterviewSessionService`
- `AnswerEvaluationService`
- `InterviewSessionCache`
- `PdfExportService`

### 实现思路
这个模块的核心不只是“生成问题”，而是 **会话生命周期管理**：

1. 创建会话时，如果已有未完成会话，优先复用，避免重复生成
2. 问题列表和当前进度先放 Redis，保证读取快、状态更新快
3. 如果 Redis 缓存丢失，再从数据库恢复问题列表和答案进度
4. 用户答题时，既更新缓存，也同步保存到数据库
5. 最后一题提交后，把评估任务丢到 Redis Stream 异步执行
6. 评估服务按批次处理问题，最后做汇总总结，生成面试报告

### 这个模块的面试亮点
- **Redis 缓存 + 数据库恢复**：兼顾性能与可靠性
- **会话状态机设计**：CREATED、IN_PROGRESS、COMPLETED、EVALUATED
- **异步评估**：把大模型耗时操作从用户主链路里拆出去
- **批量评估 + 二次汇总**：减少单次 prompt 过长导致的 token 和稳定性问题

---

## 5.3 知识库模块

### 模块职责
知识库模块负责：
- 上传知识库文件
- 提取文本内容
- 保存原文件与元数据
- 异步发起向量化
- 提供基于知识库的问答接口
- 支持多知识库联合检索
- 支持列表、分类、下载、删除、重新向量化等能力

### 关键类
- `KnowledgeBaseController`
- `KnowledgeBaseUploadService`
- `KnowledgeBaseQueryService`
- `DocumentParseService`
- `FileStorageService`

### 实现思路
1. 上传文件后先做大小、MIME、扩展名校验
2. 通过文件哈希识别重复知识库，避免重复向量化
3. 解析正文文本并保存原始文件到对象存储
4. 元数据入库，向量化状态先标记为 `PENDING`
5. 把向量化任务放进 Redis Stream，异步处理
6. 查询时按知识库 ID 限定检索范围，再用 pgvector 做相似度搜索
7. 命中有效上下文后，再把上下文拼给大模型生成最终回答

### 这个模块的面试亮点
- 不是直接“把文件喂给模型”，而是先做 **对象存储、文本提取、异步向量化、检索增强生成**
- 支持多知识库联合问答，更接近实际业务场景
- 对重复文件和失败重试都做了工程处理

---

## 5.4 RAG Chat 模块

### 模块职责
RAG Chat 模块负责：
- 创建会话
- 绑定一个或多个知识库
- 保存用户消息和助手消息
- 通过 SSE 进行流式回答
- 在流式结束后回写完整消息内容

### 关键类
- `RagChatController`
- `RagChatSessionService`
- `KnowledgeBaseQueryService`

### 实现思路
这个模块最有代表性的点是：**把“流式输出”和“消息落库”结合起来**。

实现方式是：
1. 用户发起消息时，先同步保存用户消息
2. 同时创建一条 AI 消息占位记录
3. 再发起流式 SSE 输出
4. 流式过程中不断把 chunk 推给前端
5. 流式结束或失败后，把最终内容回写到 AI 消息记录里

这样做的好处是：
- 前端体验是实时流式
- 后端数据库里又能保留完整聊天记录
- 即使中途中断，也能保留已生成的内容

## 6. 端到端业务流程

## 6.1 简历上传 → 解析 → AI 评分 → 结果保存 / 导出

### 流程说明
1. 用户上传简历文件
2. 服务端校验大小、类型、是否为空
3. 检查是否存在重复简历，若重复则优先复用历史结果
4. 使用 `DocumentParseService` 提取文本并清洗内容
5. 使用 `FileStorageService` 把原文件上传到对象存储
6. 把简历元数据和文本保存到 PostgreSQL，状态记为 `PENDING`
7. 把分析任务发送到 Redis Stream
8. 异步消费者调用 `ResumeGradingService` 做 AI 评分
9. 通过结构化输出把 AI 结果解析为业务对象并持久化
10. 用户可查看历史分析结果，或通过 `PdfExportService` 导出 PDF

### 这条链路体现的工程思路
- 把“上传成功”和“分析完成”拆开，提升接口响应速度
- 原文件与分析结果分离存储，既保留原始材料，也便于后续重分析
- 对重复简历做历史复用，降低重复调用模型成本

---

## 6.2 面试会话创建 → 问题生成 → 回答提交 → 异步评估 → 报告导出

### 流程说明
1. 用户发起创建面试请求
2. 系统先检查是否已有未完成会话，避免重复创建
3. 根据简历文本和历史问题生成新的面试题
4. 题目列表和当前进度写入 Redis
5. 同步把会话元数据写入 PostgreSQL，作为恢复依据
6. 用户每回答一题，先更新 Redis，再持久化答案
7. 当最后一题提交或提前交卷时，把评估任务放到 Redis Stream
8. `AnswerEvaluationService` 分批评估每题表现，再二次汇总生成整体评价
9. 结果保存后可查看报告，并通过 `PdfExportService` 导出 PDF

### 这条链路体现的工程思路
- 热数据放 Redis，持久数据放数据库
- 用户主链路做轻量操作，耗时评估通过异步化完成
- 会话支持恢复，避免用户刷新页面或短时故障就丢失上下文

---

## 6.3 知识库上传 → 解析 → 向量化 → 检索 → RAG 问答

### 流程说明
1. 用户上传知识库文件
2. 系统校验类型、扩展名、大小
3. 通过文件哈希识别重复知识库
4. 用 `DocumentParseService` 提取正文
5. 把原始文件上传到对象存储
6. 把知识库元数据写入 PostgreSQL，状态记为 `PENDING`
7. 把向量化任务发到 Redis Stream
8. 异步任务把知识内容写入 pgvector 向量库
9. 用户发起问答时，根据知识库 ID 做限定检索
10. 先检索，再把命中上下文交给大模型生成答案

### 这条链路体现的工程思路
- “文件管理”和“向量检索”分层清晰
- 检索与生成解耦，后续可替换 embedding 模型或检索策略
- 支持失败后重试向量化，符合实际业务需求

---

## 6.4 RAG Chat 会话创建 → 消息持久化 → SSE 流式回答 → 助手消息落库

### 流程说明
1. 用户创建一个 RAG Chat 会话，并绑定知识库
2. 用户发送问题时，先把用户消息写入数据库
3. 同时创建一条 AI 消息占位记录
4. 基于绑定知识库执行 RAG 检索
5. 通过 SSE 持续向前端推送回答片段
6. 流式结束后，把完整 AI 内容回写到消息表
7. 之后用户可以查看完整历史消息，形成连续对话

### 这条链路体现的工程思路
- 兼顾实时体验与消息留痕
- 适合继续扩展对话摘要、会话搜索、上下文压缩等能力

## 7. 关键实现方法

## 7.1 `StructuredOutputInvoker`：统一封装结构化 AI 输出

### 它解决什么问题
LLM 最大的工程痛点之一是：
**模型明明“理解了”，但输出不一定稳定、也不一定严格符合 JSON 格式。**

如果每个业务模块都自己处理重试、JSON 约束、解析失败日志，会出现：
- 代码重复
- 提示词风格不一致
- 排查问题困难

### 这个类的做法
`StructuredOutputInvoker` 做了几件关键的事：
1. 统一调用 `ChatClient`
2. 配合 `BeanOutputConverter` 做结构化输出转换
3. 当解析失败时自动重试
4. 在重试提示词中追加“必须返回合法 JSON”的强约束
5. 可选地把上一次错误原因注入提示词，帮助模型修正输出
6. 超过最大尝试次数后，统一抛业务异常

### 为什么这是亮点
这个设计体现出项目不是“把 AI 当黑盒直接调用”，而是把 AI 接口当成 **不稳定外部依赖** 来治理。

它的价值在于：
- 提高结构化输出成功率
- 降低各业务模块重复代码
- 让简历分析、面试评估等模块可以共享一套 AI 输出治理逻辑

---

## 7.2 `DocumentParseService`：文档解析做了明确的安全和性能控制

### 它解决什么问题
上传文件后，真正可用于 AI 分析和向量化的是“正文文本”，不是原始二进制文件。

如果直接粗暴解析，会遇到：
- 把图片、附件、嵌入资源一起带出来
- PDF 多栏顺序混乱
- 文本过大，导致内存或 prompt 爆炸
- 扫描版文件无法直接提取正文

### 这个类的做法
它不是直接使用 Tika 默认解析，而是做了更细的控制：
1. 使用 `AutoDetectParser` 自动识别文件类型
2. 只提取正文，不拿附件内容
3. 显式禁用嵌入文档解析，避免把图片引用、临时路径等脏内容带进来
4. 对 PDF 关闭图片提取，并按位置排序文本，改善多栏布局解析效果
5. 通过 `BodyContentHandler` 限制最大文本长度为 5MB
6. 最终再交给 `TextCleaningService` 做清洗

### 为什么这是亮点
这说明项目考虑的是“工程可用性”，不是单纯“能跑”。

对面试来说，这一段很能体现你对实际文件处理问题的理解：
- 解析不是越多越好，而是要 **提纯出最适合模型消费的文本**
- 先把输入质量管住，后面的 AI 分析才稳定

## 7.3 `FileStorageService`：对象存储管理做了通用封装

### 它解决什么问题
简历和知识库文件都需要长期保存，不能只放本地磁盘。

### 这个类的做法
1. 使用 AWS S3 SDK 对接 S3 兼容对象存储
2. 启动时自动检查 bucket 是否存在
3. 封装上传、下载、删除、存在性检查
4. 生成带日期分层的对象 key，便于后续管理
5. 给文件名加 UUID 前缀，避免命名冲突
6. 将中文文件名转为拼音并过滤特殊字符，保证对象键更安全

### 为什么这是亮点
这个设计体现了两点：
- **文件不只是“传上去”**，而是当成正式资产来管理
- 通用能力下沉后，简历模块和知识库模块都能复用同一套存储逻辑

## 7.4 `RedisService`：把 Redis 封装成统一基础设施

### 它解决什么问题
Redis 在这个项目里不是只做缓存，而是承担了三类角色：
- KV 缓存
- TTL 生命周期管理
- Stream 异步消息队列

如果每个模块都直接操作 Redisson 原始 API，业务代码会非常分散。

### 这个类的做法
1. 封装基础的 `set/get/delete/expire`
2. 提供 `getOrLoad`，减少手写缓存回源逻辑
3. 封装 Stream 消息发送
4. 支持消费者组创建
5. 支持阻塞式消费，而不是无脑轮询
6. 支持 ACK 确认处理完成

### 为什么这是亮点
这说明 Redis 在项目里被设计成了“基础平台能力”，而不是零散工具。

它给后续扩展留下了空间：
- 简历分析异步化
- 知识库向量化异步化
- 面试评估异步化
- 未来如果要做更多后台任务，也能继续复用这套能力

## 7.5 `PdfExportService`：把 AI 结果做成可交付产物

### 它解决什么问题
很多 AI 项目到最后只给一个 JSON 或网页展示，缺少“交付形态”。

而这个项目把结果进一步包装成 PDF，意味着它已经考虑：
- 用户怎么保存结果
- 用户怎么发送结果
- 用户怎么在求职场景中真正使用结果

### 这个类的做法
1. 使用 iText 8 生成 PDF
2. 内嵌中文字体，避免跨平台乱码
3. 对文本做清洗，规避特殊字符影响排版
4. 分别支持简历分析报告和模拟面试报告导出
5. 把评分、优势、建议、问答详情组织成结构化版式

### 为什么这是亮点
它体现的是“从算法结果到业务交付”的最后一公里能力。

很多项目止步于“算出来了”，这个项目进一步走到了“能导出、能复盘、能给用户用”。

## 7.6 `RateLimitAspect`：AOP + Redis Lua 实现多维度限流

### 它解决什么问题
AI 接口通常成本高、耗时长，如果不做限流，容易出现：
- 被高频请求打爆
- 模型调用费用不可控
- 恶意刷接口

### 这个类的做法
1. 使用自定义 `@RateLimit` 注解标记接口
2. AOP 统一拦截，而不是在每个 Controller 手写判断
3. 使用 Lua 脚本保证限流逻辑原子执行
4. 支持 `GLOBAL / IP / USER` 多维度限流
5. 支持 fallback 降级方法
6. 使用 Redis Hash Tag，考虑了 Cluster 模式下同槽位执行问题

### 为什么这是亮点
这不是简单地“加个 Guava 限流器”，而是更贴近分布式场景的做法。

它能说明你理解：
- AI 接口需要保护
- 多机部署时限流要放到共享中间件里做
- 原子性和一致性需要靠 Lua 保证

## 7.7 `KnowledgeBaseQueryService`：RAG 检索不是只做一次 similaritySearch

### 它解决什么问题
RAG 项目常见问题是：
- 用户问题太短，检索不到有效结果
- 检索到了弱相关文本，模型开始“胡说”
- 流式输出时，模型先输出一大段“信息不足说明”，用户体验差

### 这个类的做法
1. 支持 query rewrite，把原始问题优化后再检索
2. 针对问题长度动态调整 `topK` 和 `minScore`
3. 对短 token 做“有效命中确认”，避免把弱相关片段直接交给模型
4. 如果没有有效命中，直接返回统一兜底文案
5. 流式输出时增加探测窗口，对“无结果型长文案”做归一化处理

### 为什么这是亮点
这说明你不是把 RAG 当“向量检索 + 大模型”的公式化组合，而是意识到：
**RAG 真正的难点是检索质量控制与输出体验治理。**

---

## 7.8 `AnswerEvaluationService`：批次评估 + 二次总结，控制上下文长度与结果稳定性

### 它解决什么问题
面试题一多，如果一次性把所有问答都塞给模型：
- prompt 太长
- 成本高
- 稳定性差
- 输出结构更容易失败

### 这个类的做法
1. 按批次评估题目，默认批次大小可配置
2. 每个批次先生成问题级评估结果
3. 再合并各批次结果，补齐缺失项
4. 再做第二次总结，产出整体反馈、优势和改进建议
5. 如果二次总结失败，退回到批次聚合结果
6. 最终总分基于实际答题结果重新计算，而不是盲信 AI 原始返回值

### 为什么这是亮点
这体现的是典型的 AI 工程思维：
- 把大任务拆成小任务
- 对模型结果做多阶段治理
- 对异常场景准备降级策略

## 9. 项目不足与优化方向

这一部分面试时一定要客观，不能把项目说得“完美无缺”。能主动讲不足，反而更像真正做过工程的人。

### 9.1 自动化测试覆盖偏弱
当前测试几乎只有 `contextLoads()`，说明：
- 关键业务链路缺少单元测试
- AI 结构化输出、文档解析、限流、异步任务等都还缺专项测试
- 集成测试和回归测试体系还不完整

### 9.2 安全体系没有完整展开
从当前代码能看出：
- 限流做了
- 但用户认证、权限模型、租户隔离没有完整呈现
- `USER` 维度限流仍依赖请求属性或请求头，说明正式鉴权链路还可以补强

### 9.3 数据库 schema 管理更适合开发环境，不适合生产环境
当前配置里：
- JPA 使用 `ddl-auto: update`
- pgvector 配置 `initialize-schema: true`

这对本地开发方便，但生产环境更适合：
- 使用 Flyway / Liquibase 做数据库迁移
- 明确区分开发与生产 schema 策略

### 9.4 异步链路还可以继续增强可靠性
目前已经用了 Redis Stream，但还可以继续补：
- 消费失败重试次数控制
- 死信队列
- 任务监控面板
- 幂等与重复消费保护更明确化

### 9.5 RAG 检索能力还有继续优化空间
现在已经做了 query rewrite 和动态检索，但后续还可以继续做：
- 更细粒度 chunk 策略
- rerank 重排
- 引用来源展示
- 多轮对话上下文压缩
- 命中片段高亮

### 9.6 观测性与运维能力还不够完整
当前代码里没有看到完整的：
- Metrics 指标体系
- Trace 链路追踪
- AI 调用耗时监控
- CI/CD 流程
- 灰度与环境隔离策略

### 9.7 统一 HTTP 200 返回虽然便于前端处理，但仍有设计权衡
现在异常统一通过业务错误码返回，这种做法前后端联调会比较顺，但在更标准的 REST 风格下，也可以进一步结合更清晰的 HTTP 状态码设计。

## 10.1 30 秒版本

2. **Redis 缓存 + 数据库恢复 + Redis Stream 异步化**：把会话状态与耗时任务分离，提高性能和可恢复性
3. **RAG 检索工程化**：不仅做 similarity search，还做 query rewrite、动态 topK/minScore、无效命中识别和流式输出归一化

## 11. 高频追问与参考回答

### 11.1 为什么要单独封装 `StructuredOutputInvoker`，不用每个模块自己调 `.entity()`？
可以这样答：

> 因为 LLM 的主要问题不是“不会返回内容”，而是“返回内容不稳定”。如果每个模块都自己处理 JSON 约束、重试、日志、异常，会出现大量重复代码，而且策略不一致。把它抽出来以后，简历分析和面试评估都能复用同一套结构化输出治理逻辑，工程上更稳定。

### 11.2 为什么面试会话要先放 Redis，又要落数据库？
可以这样答：

> 因为面试会话是典型的热状态数据，频繁读取当前题目、当前进度，用 Redis 响应更快；但只放缓存不安全，所以又把关键数据同步到数据库。这样既能保证体验，也能在缓存丢失后恢复状态。

### 11.3 为什么 RAG 选 PostgreSQL + pgvector，而不是单独上向量数据库？
可以这样答：

> 这个项目规模下，优先考虑的是工程复杂度和落地效率。PostgreSQL 本来就已经承载业务数据，再加 pgvector 可以把业务数据和向量检索统一在一个数据栈里，开发和运维成本更低。后面如果规模再上去，再考虑拆专门的向量检索系统。

### 11.4 为什么要把简历分析、知识库向量化、面试评估做成异步任务？
可以这样答：

> 因为这些任务都依赖 AI 或文本处理，耗时不可控。如果强行放在同步接口里，会让用户等待时间很长，也容易超时。异步化后，主接口可以快速返回状态，后台再慢慢处理，整体体验更稳定。

### 11.5 RAG 项目里你觉得最容易被忽略的问题是什么？
可以这样答：

> 我觉得最容易被忽略的是“检索命中不等于可回答”。如果把弱相关文本直接交给模型，模型就会生成看似合理但其实不可靠的回答。所以我在检索侧增加了 query rewrite、动态阈值和有效命中判断，先控制上下文质量，再交给模型生成。

### 11.6 这个项目如果继续做生产化，你会优先补什么？
可以这样答：

> 我会优先补三块：第一是测试，包括文档解析、结构化输出、异步链路和 RAG 检索的集成测试；第二是数据库迁移和环境配置治理，把 `ddl-auto` 这类开发配置替换掉；第三是监控告警和任务观测，尤其是 AI 调用耗时、失败率和 Redis Stream 消费状态。
